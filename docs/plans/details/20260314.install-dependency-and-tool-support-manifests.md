## Goal

- [ ] Define and implement a single machine-readable dependency and compatibility model so packages, skills, commands, hooks, rules, scripts, and plugin/template entrypoints can declare what they need, what is optional, and which tools can actually support and install them.

## Non-goals

- [ ] Do not replace `docs/tools/` as the human-readable source of truth for audited vendor capability claims.
- [ ] Do not force all tools to expose the same native surface; Codex may continue to realize some workflows differently from Claude or Cursor.
- [ ] Do not collapse package composition and artifact dependency metadata into one file unless the implementation work proves that is simpler and testable.
- [ ] Do not remove existing install paths before equivalent dependency closure and validation are in place.

## Problem Statement

- [ ] MDT currently has partial requirement models in several places:
  - package manifests declare package-level `requires`
  - skills declare some runtime and companion requirements in `skill.meta.json`
  - commands declare tool targeting in `commands/*.meta.json`
  - plugin and template install surfaces declare what to install, but not the transitive dependency closure of those entrypoints
- [ ] This leaves no single way to answer:
  - what must be installed if a given package, skill, hook, command, rule, or script is selected
  - which dependencies are required vs optional
  - whether a target tool supports the required capability set well enough to allow installation

## Desired Outcome

- [ ] Every installable artifact type can declare:
  - required dependencies
  - optional dependencies
  - required runtime capabilities
  - supported tool targets
- [ ] Every tool surface has a machine-readable support map that states:
  - what MDT always installs for that tool
  - what that tool supports natively, experimentally, via repo adapter, or not at all
  - what install-time capability checks should hard-fail vs warn
- [ ] `mdt install` can compute a dependency closure and reject or warn on incompatible package selections before copying files.
- [ ] `plugin.json` or its backing metadata can represent a self-sufficient install surface instead of only top-level directories.

## Open Questions / Assumptions

- [ ] Whether dependency metadata should live beside each artifact type (`command.meta.json`, `hook.meta.json`, `rule.meta.json`, `script.meta.json`) or in aggregated per-directory indexes.
- [ ] Whether `.claude-plugin/plugin.json` should directly list dependency closure or be generated from the new metadata model.
- [ ] Whether package manifests should declare only top-level intent and let dependency resolution expand the closure automatically.
- [ ] Whether optional dependencies should support tool-conditional activation, for example the Codex ai-learning observer.
- [ ] Whether tool support should be defined in `docs/tools/` source data and exported for installers, or in a dedicated machine-readable config with docs generated from it.

## Decisions

- [ ] Separate two concepts:
  - dependency closure for what an artifact needs
  - tool support/capability for what a target can install and run
- [ ] Preserve `docs/tools/` as human-readable audited documentation, while adding a machine-readable tool support map for installer logic.
- [ ] Treat required dependencies as install blockers and optional dependencies as capability-gated enhancements.
- [ ] Keep package manifests as the user-facing composition layer; they may reference artifacts without needing to duplicate every transitive dependency manually.

### Workstream: model tool support maps

- Status: `not-started`
- [ ] Add per-tool machine-readable support maps for Claude, Cursor, and Codex.
- [ ] Record for each tool:
  - always-installed baseline surfaces
  - supported artifact types

  - whether a mismatch should error or warn at install time
- [ ] Ensure the support maps align with `docs/supported-tools.md` and `docs/tools/capability-matrix.md`.
- [ ] Add validation to catch drift between the machine-readable support maps and the audited tool docs.

- Status: `not-started`
