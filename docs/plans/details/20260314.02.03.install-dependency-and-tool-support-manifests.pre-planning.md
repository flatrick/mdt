## Purpose

This document captures **missing details (gaps)** and **risks** identified for the Install Dependency and Tool Support Manifests work. Its focus is **filling out those gaps and addressing the shortcomings/risks** so that the main plan can be implemented without guesswork and with known mitigations in place.

Use this doc when working from any machine or in a team: ensure each gap is resolved in the main plan or discovery plan before implementation, and each risk has an agreed mitigation reflected in the main plan or in code/docs.

---

## Missing details (gaps to fill)

### 1. Schema and vocabulary not specified

The plan commits to a "shared dependency sidecar schema" and a "common vocabulary" (required/optional deps, runtime capabilities, supported tool targets, target-shaped add-ons, generated install surfaces) but does not define:

- **Field names and types** for sidecars (e.g. `requires`, `optional`, `capabilities`, `tools`).
- **Capability identifiers** — Today `scripts/install-mdt.js` uses `TARGET_CAPABILITIES` with `hooks`, `runtimeScripts`, `sessionData`. The plan does not list the full capability set or how it maps to package `requires` and the new support maps.
- **Version or constraint semantics** — Whether dependencies can express version ranges or only presence.

**Recommendation:** Add a "Schema sketch" subsection under workstream 1 with a minimal JSON example and the explicit list of capability IDs before implementation.

- [ ] Gap filled in main plan or [discovery plan](20260314.02.04.install-schema-and-location-discovery.md).

### 2. "Load-bearing" and "install-relevant" entrypoints undefined

The plan repeatedly uses "load-bearing entrypoints" and "install-relevant runtime entrypoints" for sidecar scope ([main plan](20260314.02.03.install-dependency-and-tool-support-manifests.md) lines 49, 62, 102–104). There is no:

- Definition of what makes an entrypoint "load-bearing" or "install-relevant".
- Enumerated list or discovery rule (e.g. "any script referenced by package `tools.*.scripts` or by a hook").

Today the installer hardcodes entrypoints (e.g. `BASELINE_SHARED_SKILL_NAMES`, `TARGET_CAPABILITIES` in `scripts/install-mdt.js`). Moving to a file-driven model requires a clear rule so implementers and validators know which files get sidecars.

**Recommendation:** Define criteria in workstream 1 (e.g. "referenced by a package manifest, a hook, or a tool support map") and list the first vertical slice entrypoints explicitly (e.g. `scripts/codex-observer.js`, `skills/ai-learning/scripts/*.js`, etc.).

- [ ] Gap filled in main plan or [discovery plan](20260314.02.04.install-schema-and-location-discovery.md) question 3.

### 3. `metadata/tools/` layout and support-map schema

- **Location:** Plan says machine-readable maps live under `metadata/tools/` but the repo currently has **no** `metadata/` directory.
- **File layout:** Not specified — one file per tool (`claude.json`, `cursor.json`, `codex.json`)? One combined matrix? How are "always-installed baseline surfaces" and "supported artifact types" split across files?
- **Support-map schema:** "Record for each tool: always-installed baseline surfaces, supported artifact types, capability status by feature (official, experimental, repo-adapter, unsupported), error vs warn" — no field names or example structure.

**Recommendation:** Add a concrete file-naming and single-file schema example in workstream 2 (or workstream 1) so validators and installer can be implemented without guesswork.

- [ ] Gap filled in main plan or [discovery plan](20260314.02.04.install-schema-and-location-discovery.md) question 2.

---

### 4. Merge and precedence rules

Workstream 1 says "Document merge and precedence rules when package manifests, artifact sidecars, tool support maps, and per-tool add-ons all apply" but the plan does not sketch them. Open questions:

- Does package-level `requires` override or merge with skill sidecar `requires`?
- When a tool support map says "unsupported" for a capability and a package marks it required, does the package win (fail install) or the map (already implied)?
- How do template-root add-ons merge with canonical source (e.g. Codex overlay skills vs shared `skills/ai-learning`)?

**Recommendation:** Add a short "Precedence" subsection: e.g. "required = union across closure; optional = union; tool support map is authoritative for 'can install'; package/sidecar is authoritative for 'wanted'."

- [ ] Gap filled in main plan or [discovery plan](20260314.02.04.install-schema-and-location-discovery.md) question 4.

### 5. Config-fragment and template add-on contracts

- **Config-fragment contract:** "Tool-varying defaults use explicit config sidecars/fragments" — no format, location, or merge behavior (e.g. fragment filename pattern, how they are merged into tool config).
- **Template-root add-on contract:** "Per-tool add-ons stay in template roots and are limited to metadata, docs, and config shaping. They must not carry duplicate runtime trees." — no schema for what can appear there or how the installer validates "metadata/docs/config only" (e.g. allowlist of extensions or directories).

**Recommendation:** In workstream 1, add one paragraph each: config-fragment file/location and merge rule; add-on directory layout and a simple allowlist/validation rule so the first vertical slice (e.g. Codex observer, ai-learning config defaults) can be implemented consistently.

- [ ] Gap filled in main plan or [discovery plan](20260314.02.04.install-schema-and-location-discovery.md) questions 5–6.

### 6. Resolver algorithm and dry-run output

- **Resolver:** "Compute install closure from selected packages, artifact sidecars, tool baselines, and template-root add-ons" — order of expansion (packages first? depth-first? tool baseline merged when?) and handling of optional vs required (when to fail vs warn) are not written down.
- **Dry-run:** "Dry-run output explains why a package can or cannot be installed and which optional enhancements were skipped" — no structure (e.g. machine-readable vs human-only, exit codes, or format).

**Recommendation:** Add a short "Resolver order" and "Dry-run contract" (e.g. exit 0 + summary when ok; non-zero + clear reason when required unsatisfied; optional-skipped listed in stdout or a structured block).

- [ ] Gap filled in [main plan](20260314.02.03.install-dependency-and-tool-support-manifests.md) resolver workstream.

### 7. Generated install surfaces

"Generate plugin/template install surfaces from the resolved model where practical" is underspecified:

- **Which surfaces?** e.g. `.claude-plugin/plugin.json`, template file lists, both?
- **Format** of generated output and where it is written.
- **"Where practical"** — criteria to generate vs keep hand-maintained (e.g. "generate plugin.json from closure; keep template dir layout hand-maintained for now").

The plan already allows splitting this as a follow-up; without a clear boundary, scope creep is likely.

**Recommendation:** Name the exact artifacts to generate in this plan (e.g. "plugin.json contents" only) and state that template directory layout remains hand-maintained unless a follow-up plan extends it.

- [ ] Gap filled in [main plan](20260314.02.03.install-dependency-and-tool-support-manifests.md).

### 8. Migration scope and backward compatibility

- **Migration:** "Migrate existing packages, skills, commands, hooks, rules, and scripts to the new metadata model" — no count of artifacts, no order of migration (after first vertical slice?), and no rollback or feature-flag strategy.
- **Backward compatibility:** Current installer behavior is driven by hardcoded constants. Switching to file-driven closure might change what gets installed for existing `mdt install` invocations. The plan says "Do not remove existing install paths before equivalent dependency closure... are in place" but does not define "equivalent" or how to verify no regression.

**Recommendation:** Add a migration subsection: (1) list artifact counts (e.g. packages, skills with skill.meta.json, commands with .meta.json); (2) state that existing install behavior is preserved until migration is complete and tests lock it; (3) optionally add a compatibility flag (e.g. read old manifest shape if sidecar missing) with a sunset version.

- [ ] Gap filled in [main plan](20260314.02.03.install-dependency-and-tool-support-manifests.md).

### 9. First vertical slice enumeration

The first vertical slice names "continuous-learning, ai-learning, optional Codex observer" but does not enumerate:

- Which exact files get dependency sidecars (e.g. which scripts under `skills/ai-learning/`, `scripts/codex-observer.js`).
- Which config fragments exist and for which tools.
- How "optional Codex observer" is expressed in the new model (e.g. optional package, or optional dependency of continuous-learning).

**Recommendation:** Add a small "First slice artifact list" under that workstream: sidecar locations, one or two example config fragments, and the optional-dependency wiring for the observer.

- [ ] Gap filled in [main plan](20260314.02.03.install-dependency-and-tool-support-manifests.md) first vertical slice workstream or [discovery plan](20260314.02.04.install-schema-and-location-discovery.md).

---

### 10. Validation and CI drift checks

- "Add validation to catch drift between the machine-readable support maps and the audited tool docs" — mechanism is not specified (e.g. automated comparison of `docs/tools/capability-matrix.md` vs `metadata/tools/*.json`, or manual checklist).
- Schema validators: no mention of schema versioning or backward-compatible evolution.

**Recommendation:** Specify that drift check is an automated script comparing support map contents to a defined subset of docs (e.g. capability-matrix table) and failing CI on diff; and add a note that support-map and sidecar schemas include a version field for future compatibility.

- [ ] Gap filled in [main plan](20260314.02.03.install-dependency-and-tool-support-manifests.md) validation workstream.

## Risks to address

### R1. Scope creep and dependency on "generated" surfaces

Six workstreams (schemas, tool maps, artifact metadata, resolver, first slice, validation, migration) are interdependent. "Generate plugin/template install surfaces" is left partly to "where practical," which can grow. The plan allows a follow-up split but does not define the boundary.

**Mitigation:** Lock the scope of "generated" in this plan (e.g. generated plugin.json only; template layout out of scope) and add an explicit "Out of scope for this plan" line.

- [ ] Mitigation reflected in [main plan](20260314.02.03.install-dependency-and-tool-support-manifests.md) or implementation.

---

### R2. Breaking existing install behavior

`scripts/install-mdt.js` uses `TARGET_CAPABILITIES` and many `BASELINE_*` constants. Moving to file-driven closure and support maps can change what gets installed for current packages (e.g. continuous-learning, language packages). Users who rely on current behavior may see different file sets or failures.

**Mitigation:** Define "equivalent closure" (e.g. same set of skills/rules/commands/scripts for the same package + tool) and add tests that compare current installer output (or a snapshot) to resolver output before switching; keep a fallback path until migration is complete.

- [ ] Mitigation reflected in main plan or implementation.

---

### R3. Doc/source-of-truth drift

Machine-readable maps under `metadata/tools/` must stay aligned with `docs/tools/capability-matrix.md` and `docs/supported-tools.md`. The plan says CI validates but not how. If the process is manual, drift is likely.

**Mitigation:** Implement an automated drift check (see missing detail #10) and run it in CI; document in the plan.

- [ ] Mitigation reflected in [main plan](20260314.02.03.install-dependency-and-tool-support-manifests.md) or implementation.

---

### R4. Blocking ai-learning and observer flows

[AI-Learning Self-Contained](20260314.12.04.ai-learning-self-contained.md) is **halted** on this plan. Slip here directly delays the ai-learning vertical slice and any flows that depend on the optional Codex observer.

**Mitigation:** Keep the first vertical slice workstream tightly scoped (continuous-learning, ai-learning, Codex observer only) and sequence it so resolver + support maps are usable before full migration; document the unblock criteria in both plans.

- [ ] Mitigation reflected in main plan and ai-learning plan.

---

### R5. Compose redesign dependency

The [Composable MDT Redesign](20260314.12.04.composable-mdt-redesign.md) is expected to consume this plan's schema and resolver. If this plan's contract changes later (e.g. new capability types, different sidecar shape), the compose work may need rework.

**Mitigation:** Add a version field to the schema and support maps from day one; document a short "stability contract" (e.g. additive-only changes for N releases) in this plan or in a shared design doc.

- [ ] Mitigation reflected in main plan or implementation.

---

### R6. Optional dependency UX and exit codes

The plan decides "warn, but continue" for unsupported optional dependencies. Undefined: how warnings are presented (stderr, log file), whether exit code remains 0, and how `--dry-run` represents optional-skipped vs required-failed.

**Mitigation:** Define in the plan: exit code 0 when all required satisfied (with optional skipped); non-zero when any required unsatisfied; warnings to stderr with a consistent prefix; dry-run clearly separates "would skip (optional)" from "would fail (required)."

- [ ] Mitigation reflected in [main plan](20260314.02.03.install-dependency-and-tool-support-manifests.md).

---

### R7. Cycle detection and extends

Validation workstream mentions "cycle detection for artifact dependency graphs." Package `extends` (see `docs/package-manifest-schema.md`) already rejects cycles. Unclear: whether the new graph includes skills, commands, and sidecar-declared dependencies, and how cycles are reported.

**Mitigation:** Specify that the dependency graph includes packages (extends), package→skills/rules/commands, and sidecar-declared requires; define one cycle-detection pass and a single, clear error format.

- [ ] Mitigation reflected in [main plan](20260314.02.03.install-dependency-and-tool-support-manifests.md) validation workstream.

---

### R8. Enforcement of "add-ons = metadata/docs/config only"

"Template roots must not carry duplicate runtime trees" is a design rule but the plan does not say how it is enforced (installer check, CI validator, or review-only). Accidental runtime code in add-ons could reintroduce the duplication the plan aims to avoid.

**Mitigation:** Add a validator or installer step that fails if template add-on directories contain disallowed entrypoints (e.g. `.js` under add-on skill dirs beyond an allowlist) and reference it in the validation workstream.

- [ ] Mitigation reflected in [main plan](20260314.02.03.install-dependency-and-tool-support-manifests.md) validation workstream.

## Summary

| Area                    | Missing detail                                | Risk                          |
| ----------------------- | --------------------------------------------- | ----------------------------- |
| Schema                  | Field names, capability IDs, no version field | Rework, compose drift (R5)    |
| Entrypoints             | "Load-bearing" / "install-relevant" undefined | Inconsistent sidecar coverage |
| metadata/tools/         | File layout and support-map schema            | Implementation guesswork      |
| Merge/precedence        | Package vs sidecar vs tool map                | Resolver ambiguity            |
| Config/add-on contracts | Format, location, validation                  | First slice inconsistency     |
| Resolver / dry-run      | Order, exit codes, output shape               | UX and scriptability          |
| Generated surfaces      | Which artifacts, "practical" boundary         | Scope creep (R1)              |
| Migration               | Counts, order, backward-compat               | Breaking installs (R2)        |
| First slice             | Exact sidecars and fragments                  | ai-learning block (R4)        |
| Drift checks            | Automated vs manual                           | Doc drift (R3)                |
| Optional deps           | Warnings, exit codes                          | UX (R6)                       |
| Cycles                  | Graph scope, reporting                        | Validation gaps               |
| Add-on enforcement      | How to block runtime in add-ons               | Duplicate trees (R8)          |

**Suggested order:** Address the schema and entrypoint definitions, support-map layout, and resolver/dry-run contract first (workstream 1 and start of resolver workstream), then lock migration and backward-compat and the first-slice artifact list so the ai-learning plan can unblock predictably.
