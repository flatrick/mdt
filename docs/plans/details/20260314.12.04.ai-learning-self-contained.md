## Goal

Make the `ai-learning` skill fully self-contained by moving its runtime
dependencies from `scripts/lib/continuous-learning/` into the skill directory.
This eliminates the cross-layer `require` chains, fixes all four bugs found on
2026-03-14, and makes the skill work identically in the repo, in tests, and
after install.

`ai-learning` is the single most important capability of MDT. It is currently
broken: `mdt learning analyze` silently times out, the analysis prompt hangs
`claude --print`, and the path resolution depends on fragile fallback chains.

## Problem Summary

Four bugs were discovered during a `/ai-learning` session on 2026-03-14:

### Bug 1 — Silent 60-second timeout on `mdt learning analyze`

`mdt.js` spawns `codex-learn.js` via `spawnSync` with `timeoutMs: 60_000`.
`codex-learn.js analyze` in turn spawns the AI tool asynchronously and `await`s
it. The outer `spawnSync` kills the process after 60 seconds; stdout and stderr
are both empty, exit code 1. The user sees nothing and cannot diagnose the
failure.

### Bug 2 — `buildLearningBasicCommand` overrides the correct CWD default

`runNodeScript` defaults to `cwd: REPO_ROOT` (the install root). Every learning
command builder overrides this with `cwd: options.cwd || process.cwd()`. When no
`--cwd` flag is given, the calling shell's CWD is used instead of the install
root, making the commands sensitive to where the caller happens to be.

### Bug 3 — `claude --print` hangs waiting for tool-use approval

`buildAnalysisPrompt` produces `"Read <observationsFile> and identify patterns..."`.
This causes `claude --print --max-turns 3` to invoke the Read tool. In `--print`
mode without `--dangerously-skip-permissions`, tool calls block on interactive
approval that never arrives. All six `observer.log` entries from this session
show "Analyzing..." with no completion — every analysis invocation since 10:23
UTC has been hanging indefinitely.

### Bug 4 — Multi-candidate path resolution chains are fragile

`detect-project.js` and `codex-learn.js` each have a hand-rolled `loadXModule()`
that tries 2–4 candidate paths. Any directory layout change silently breaks the
chain. Root cause: scripts do not use `__dirname`-relative paths consistently.

The installed layout adds an `mdt/` prefix (`~/.claude/mdt/scripts/lib/`) that
the repo layout does not have (`scripts/lib/`). The wrapper scripts bridge this
gap with fallback chains where only one candidate ever resolves — the rest are
dead code:

```
Candidate 1: ~/.claude/scripts/lib/continuous-learning/     ← NEVER exists
Candidate 2: ~/.claude/skills/scripts/lib/...               ← NEVER exists
Candidate 3: ~/.claude/mdt/scripts/lib/continuous-learning/ ← ACTUALLY WORKS
Candidate 4: ~/.claude/skills/mdt/scripts/lib/...           ← NEVER exists
```

### Root cause

The current architecture tries to be clever about where things are and which
tool is active, at the expense of simplicity and resilience. Tool-specific
concerns (how to invoke `claude`, `cursor agent`, `codex`) are mixed into
the shared runtime. The skill's runtime dependencies live in a different
directory tree (`mdt/scripts/lib/`) than the skill itself (`skills/ai-learning/`),
requiring cross-layer path resolution that breaks silently.

## Decisions

- **Self-contained skill**: move `scripts/lib/continuous-learning/*.js` into
  `skills/ai-learning/lib/`. Replace all multi-candidate `require` chains with
  single `require('../lib/...')` calls.
- **Rename `codex-learn.js`** to `learn-cli.js` — it is no longer
  Codex-specific; it is the shared entry point for all explicit `mdt learning`
  invocations.
- **No tool calls in the analysis prompt**: read the observations file in Node
  and pass content inline. The AI model never needs to call Read or any other
  tool.
- **Fix the `mdt.js` timeout**: either stream the child process or remove the
  60-second cap for analyze.
- **Explicit workspace context**: remove the `|| process.cwd()` fallback from
  learning command builders. Pass workspace via `MDT_WORKSPACE` env var or
  `--cwd` flag.
- **Per-tool invocation logic** stays in `observer-runtime.js` for now but moves
  with it into the skill. Splitting into per-tool drivers is a follow-up.
