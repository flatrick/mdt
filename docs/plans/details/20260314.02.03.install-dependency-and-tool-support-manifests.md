## Goal

- [ ] Define and implement a single machine-readable dependency and tool-support model so packages, skills, commands, hooks, rules, scripts, and install-relevant runtime entrypoints can declare what they need, what is optional, and which tools can actually support and install them.

## Why This Comes First

- [ ] This plan is the prerequisite for the `ai-learning` self-contained work.
- [ ] `ai-learning` plus the optional Codex observer is the first end-to-end vertical slice this plan must enable.
- [ ] The future compose redesign must consume this plan's contract instead of inventing a parallel dependency model.

## Non-goals

- [ ] Do not replace `docs/tools/` as the human-readable source of truth for audited vendor capability claims.
- [ ] Do not force all tools to expose the same native surface; Codex may continue to realize some workflows differently from Claude or Cursor.
- [ ] Do not implement the compose redesign in this plan.
- [ ] Do not remove existing install paths before equivalent dependency closure, target-shaping, and validation are in place.
- [ ] Do not introduce per-tool runtime forks as part of the metadata model.

## Problem Statement

- [ ] MDT currently has partial requirement models in several places:
  - package manifests declare package-level `requires`
  - skills declare some runtime and companion requirements in `skill.meta.json`
  - commands declare tool targeting in `commands/*.meta.json`
  - plugin and template install surfaces declare what to install, but not the transitive dependency closure of those entrypoints
- [ ] This leaves no single way to answer:
  - what must be installed if a given package, skill, hook, command, rule, or script is selected
  - which dependencies are required vs optional
  - whether a target tool supports the required capability set well enough to allow installation
  - how a canonical repo source should become a target-shaped install output without duplicating runtime trees
- [ ] The current `ai-learning` work surfaced questions about optional hooks, optional observer add-ons, config defaults, and generated install surfaces that should be solved generically here rather than locally in one skill plan.

## Decisions

- [x] Separate two concepts:
  - dependency closure for what an artifact needs
  - tool support/capability for what a target can install and run
- [x] Machine-readable tool support maps live under `metadata/tools/`.
- [x] `docs/tools/` remains the human-readable audited source of truth; installer logic reads the machine-readable maps and CI validates the docs against them.
- [x] Dependency metadata uses co-located sidecars.
- [x] Sidecars cover load-bearing entrypoints and install-relevant runtime entrypoints, not every internal helper file.
- [x] Package manifests remain the top-level composition layer; they express user-facing install intent and let the resolver expand transitive closure automatically.
- [x] MDT may install target-shaped outputs from one canonical source.
- [x] Per-tool add-ons stay in template roots and are limited to metadata, docs, and config shaping. They must not carry duplicate runtime trees.
- [x] Tool-varying defaults use explicit config sidecars/fragments rather than full copied artifact trees.
- [x] Required dependencies and unsupported required capabilities hard-fail installs. Optional dependencies become capability-gated enhancements with clear warnings.
- [x] Plugin/template install surfaces should be generated from the dependency/support model where practical.
- [x] The optional Codex observer remains a separate optional package/add-on.
- [x] `ai-learning` is the first end-to-end vertical slice this model must support.
- [x] Per-tool override blocks use **key-level replacement**: if `tools.codex` is present, each top-level key it contains (`requires`, `capabilities`) replaces the corresponding default key entirely; keys absent from the override inherit the default array unchanged.
  - Example A: `tools.codex` contains only `capabilities` → default `requires` array used unchanged; codex `capabilities` array replaces default.
  - Example B: `tools.codex` contains only `requires` → codex `requires` replaces default; default `capabilities` array used unchanged.
  - Example C: a dependency absent from the resolved array for a tool is not evaluated (neither required nor optional).
- [x] Install fail hierarchy for capability requirements:
  - `required` + `unsupported` → **hard fail** (always)
  - `required` + `experimental` → **fail unless `--experimental` flag** is passed (non-zero exit; distinct user message from "capability absent")
  - `optional` + `unsupported` or `experimental` → **warn + skip**
- [x] Capability entries in sidecar schema support `optional: true`; absent `optional` means required (subject to fail hierarchy above).
- [x] Generated install surfaces (e.g. `plugin.json` generation from dependency closure) are **out of scope** for this plan. Template directory layout remains hand-maintained. Defer to a follow-up plan after the resolver is stable.
- [x] `--new-resolver` flag gates the **entire install path for all packages**; default is the old hardcoded path until migration is complete.
- [x] `featureStatus.mcp` uses the same status vocabulary as other capabilities. The installer **warns but never hard-fails** for MCP capability requirements regardless of `featureStatus.mcp` value; MCP is instance-dependent (users configure their own servers).

## Deliverables

- [ ] A shared dependency sidecar schema for packages, skills, commands, hooks, rules, scripts, and install-relevant runtime entrypoints.
- [ ] Machine-readable tool support maps under `metadata/tools/`.
- [ ] Installer resolution logic that computes required closure, optional enhancements, target-shaped outputs, and generated install surfaces.
- [ ] Validators and CI checks for metadata correctness and docs drift.
- [ ] A migrated first vertical slice covering `ai-learning` (renamed from `continuous-learning`) and the optional Codex observer.

### Workstream: resolve dependency closure in installer

- Status: `not-started`
- [ ] Add a dependency resolver to `scripts/install-mdt.js`.
- [ ] Compute install closure from selected packages, artifact sidecars, tool baselines, and template-root add-ons.
- [ ] Generate target-shaped outputs from canonical repo sources plus metadata/docs/config add-ons.
- [ ] Reject installations when required dependencies cannot be satisfied on the target tool.
- [ ] Warn, but continue, when only optional dependencies are unsupported.
- [ ] Ensure dry-run output explains why a package can or cannot be installed for a given tool and which optional enhancements were skipped.
- [ ] Resolver order: (1) packages → (2) transitive sidecar closure → (3) tool baseline → (4) per-tool add-ons.
- [ ] Exit code contract: exit 0 when all required capabilities satisfied (optional-skipped is fine); non-zero when any required capability unsatisfied or when required+experimental without `--experimental` flag.
- [ ] Dry-run output: human-readable stdout by default; `--format=json` emits structured JSON output.
- [ ] `--new-resolver` flag gates the entire install path for all packages; keep old hardcoded path as default until migration is complete.
