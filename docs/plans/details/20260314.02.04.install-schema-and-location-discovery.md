## Questions to Answer

### 1. Dependency sidecar schema

- [ ] What are the **field names and types** for the common dependency sidecar (e.g. `requires`, `optional`, `capabilities`, `tools`)?
- [ ] What **capability identifiers** exist? (Today the installer uses `hooks`, `runtimeScripts`, `sessionData`; the capability matrix uses `official`, `experimental`, `repo-adapter`, `unsupported`. How do these map into a single vocabulary?)
- [ ] Do dependencies express **version or constraint semantics**, or only presence?
- [ ] Should the schema include a **version field** for future compatibility?
- [ ] **Deliverable:** Schema sketch (e.g. JSON example) and a list of capability IDs that the support maps and package `requires` will use.

**Answer (Q1 ÔÇö discovery):**

- **Vocabulary (v0.0.1 minimal):** `rule`, `skill`, `runtimeScripts`, `sessionData`, `hooks`, `tools`. Add `command`, `agent`, `memory`, `mcp` when needed. MDT-owned; initial reference Claude Code; map Cursor and Codex against it (may evolve).
- **Entry shape:** Every dependency entry has at most four keys: **`type`** (required), **`optional`** (boolean; default = mandatory), **`id`** (optional; identifier for refs or MCP server), **`params`** (optional; type-specific options, e.g. hooks `mode`/`tools`, tools list). No type-specific top-level keys; type-specific data only in `params`.
- **Two top-level arrays (B's separation):** **`requires`** = which MDT things must be present (rules, skills, commands). **`capabilities`** = what the tool must support (hooks, runtimeScripts, sessionData, tools, mcp). Resolver traverses closure and checks installability on the chosen tool.
- **Per-tool overrides (NG1):** Optional **`tools`** object: keys = tool id (`claude`, `cursor`, `codex`); value = `{ requires?, capabilities? }` (arrays in same entry shape). For tool T, resolver uses **key-level replacement**: each top-level key present in `tools[T]` replaces the corresponding default key entirely; keys absent from the override inherit the default array unchanged. No array union within a key.
  - Example A: `tools.codex` contains only `capabilities` Ôćĺ default `requires` array used unchanged; codex `capabilities` array replaces default.
  - Example B: `tools.codex` contains only `requires` Ôćĺ codex `requires` replaces default; default `capabilities` array used unchanged.
  - Example C: a dependency absent from the resolved array for a tool is not evaluated (neither required nor optional).
- **Capability `optional: true` (NG2├Śschema):** Capability entries support `optional: true`. An optional capability triggers warn+skip if unsupported or experimental. Absent `optional` means required (subject to the NG2 fail hierarchy: required+unsupported Ôćĺ hard fail; required+experimental Ôćĺ fail unless `--experimental`).
- **Capability IDs:** `hooks`, `runtimeScripts`, `sessionData`, `tools`. Support map carries per-tool status for each. **MCP:** optional/mandatory MCP servers that can reduce or improve capabilities; schema and support map account for MCP as a capability/dependency that can gate/enhance others.
- **Version/constraint:** Dependencies = presence only (no version ranges). Schema **`version`** = first version **v0.0.1**; increment (e.g. v0.0.2) whenever the schema changes.

**Schema sketch (v0.0.1, updated with NG1/NG2├Śschema):**

```json
{
  "version": "0.0.1",
  "requires": [
    { "type": "rule", "id": "common/testing.md" },
    { "type": "skill", "id": "documentation-steward" },
    { "type": "skill", "id": "optional-skill", "optional": true }
  ],
  "capabilities": [
    { "type": "runtimeScripts" },
    { "type": "sessionData" },
    { "type": "hooks", "optional": true, "params": { "mode": "optional", "tools": ["claude", "cursor"] } },
    { "type": "tools", "params": { "tools": ["claude", "cursor", "codex"] } },
    { "type": "mcp", "id": "some-server", "optional": true }
  ],
  "tools": {
    "codex": {
      "requires": [
        { "type": "rule", "id": "common/testing.md" },
        { "type": "skill", "id": "documentation-steward" }
      ],
      "capabilities": [
        { "type": "runtimeScripts" },
        { "type": "sessionData" },
        { "type": "tools", "params": { "tools": ["codex"] } }
      ]
    },
    "cursor": {
      "capabilities": [
        { "type": "hooks", "params": { "mode": "optional", "tools": ["cursor"] } }
      ]
    }
  }
}
```

Note on `tools.codex` above: since `tools.codex` specifies both `requires` and `capabilities`, both keys replace their defaults when resolving for Codex. Default `requires` and `capabilities` arrays are not used.

### 2. Tool support-map schema and file layout

- [ ] What is the **file layout** under `metadata/tools/`? (One file per tool? One combined file? How are baselines vs per-feature status split?)
- [ ] What is the **support-map schema**? (Field names for ÔÇťalways-installed baseline surfaces,ÔÇŁ ÔÇťsupported artifact types,ÔÇŁ ÔÇťcapability status by feature,ÔÇŁ ÔÇťerror vs warn on mismatch.ÔÇŁ)
- [ ] How does the schema align with the existing vocabulary in `docs/tools/capability-matrix.md` (e.g. official, experimental, repo-adapter, unsupported)?
- [ ] **Deliverable:** File naming convention, directory layout, and a minimal JSON schema or example for one tool.

**Answer (Q2 ÔÇö discovery):**

- **File layout:** One file per tool: `metadata/tools/claude.json`, `metadata/tools/cursor.json`, `metadata/tools/codex.json`. Create the `metadata/tools/` directory when creating these files.
- **Baseline vs per-feature (Option A):** Use separate structures so one section can gain new fields without polluting the other. **`baseline`** = plain object keyed by type: `{ "skills": string[], "commands": string[], "rules"?: string[] }` ÔÇö what is always installed for this tool. **`featureStatus`** = object mapping each MDT artifact/capability type (from Q1 vocabulary) to `{ "status": "official" | "experimental" | "repo-adapter" | "unsupported", "toolRealization"?: string }` ÔÇö resolver uses this to decide support level when a dependency requires that type. Unsupported + required Ôćĺ fail install; experimental + required Ôćĺ warn (per parent plan). Status values align with `docs/tools/capability-matrix.md`.

**Schema sketch ÔÇö example for Cursor (`metadata/tools/cursor.json`):**

```json
{
  "baseline": {
    "skills": ["api-design", "documentation-steward", "coding-standards"],
    "commands": ["docs-audit.md", "install-rules.md"]
  },
  "featureStatus": {
    "rule": { "status": "official", "toolRealization": ".cursor/rules/" },
    "skill": { "status": "official", "toolRealization": "~/.cursor/skills/" },
    "hooks": { "status": "experimental" },
    "runtimeScripts": { "status": "official" },
    "sessionData": { "status": "official" },
    "tools": { "status": "official" },
    "mcp": { "status": "official" }
  }
}
```

**NG3 ÔÇö MCP featureStatus exception:** `featureStatus.mcp` uses the same `official`/`experimental`/`repo-adapter`/`unsupported` vocabulary as all other capability types. However, the installer **warns but never hard-fails** for MCP capability requirements, regardless of the `featureStatus.mcp` value. Rationale: MCP is instance-dependent ÔÇö users configure their own MCP servers and MDT cannot know at install time which servers are available. Document this exception explicitly in tool support map validators and installer logic.

### 3. Load-bearing and install-relevant entrypoints

- [ ] What **criteria** define ÔÇťload-bearingÔÇŁ vs ÔÇťinstall-relevantÔÇŁ runtime entrypoints? (e.g. ÔÇťreferenced by a package manifest, a hook, or a tool support map.ÔÇŁ)
- [ ] Is there a **discovery rule** (e.g. ÔÇťany script in package `tools.*.scripts` or invoked by a hookÔÇŁ) or an **enumerated list** for the first vertical slice?
- [ ] Which **exact entrypoints** in the first vertical slice (ai-learning package/skill, optional observer add-on) must have sidecars? (List or rule so implementers know what to add.)
- [ ] **Deliverable:** Short definition or rule set, plus an explicit list of first-slice entrypoints that need sidecars.

**Answer (Q3 ÔÇö discovery):**

- **Criteria (Option D):** An entrypoint is **load-bearing** (and thus **install-relevant**) if and only if it is referenced by a package manifest (in `rules`, `commands`, `skills`, or `tools.<tool>.scripts`), by a hook configuration, or by a tool support map (e.g. baseline lists); only such entrypoints receive dependency sidecars. For sidecar scope, "load-bearing" and "install-relevant" are the same.
- **Discovery rule:** Referenced by package manifest, hook, or tool support map. No transitive script closure for v0 (scripts only invoked by other scripts are not load-bearing unless something else references them).
- **First vertical slice:** Provide an explicit list so implementers know exactly which entrypoints get sidecars; the list validates the rule. Enumerate: (1) packages: ai-learning; (2) skills: ai-learning (canonical + codex overlay); (3) commands referenced by package; (4) scripts: e.g. `scripts/codex-observer.js`, `codex-template/skills/ai-learning/scripts/*.js`, and any script referenced by hooks or package `tools.codex.scripts` when reintroduced; (5) hooks that reference scripts. Document which of these must have sidecars before implementation.

**Deliverable:** Short definition (above) + explicit first-slice entrypoint list (with paths) that need sidecars.

### 4. Merge and precedence rules

**Answer (Q4 ÔÇö discovery, partial):**

- **Package vs sidecar requires:** A (union). Required = union across closure; optional = union; package and sidecar both express wanted intent.
- **Tool map vs required:** A (fail when unsupported + required). Map is authoritative for installability; package/sidecar for intent. When a required capability has **experimental** support on the tool: inform the user and require `--experimental` to install (no install without the flag).
- **Template add-on merge:** A (file-level overlay by path + allowlist). Add-on files override or supplement canonical by path; allowlist: metadata, docs, config only in add-ons; no full runtime trees. **Note (future):** Goal is to support some form of "diff" to apply rather than replacing whole files; document this as the intended direction for a later change.

### 5. Config-fragment contract

**Answer (Q5 ÔÇö format):** We cannot select a single fragment format (JSON vs TOML) for all tools: each tool has its own main config file format. Each tool will require **unique handling** of that tool's config file (format, location, and merge behavior are per-tool). Document the per-tool contract so the installer can implement tool-specific config handling.

**Answer (Q5 ÔÇö location):** Each LLM-tool (Cursor, Claude, Codex, etc.) keeps its own config file(s) in its respective `{tool}-template/` folder for simplicity. Keep config files **co-located with their partner** (e.g. skill-level config next to the skill in that template, tool-level config in the template root) rather than under `metadata/` ÔÇö so navigation stays clear and the relationship between config and artifact is obvious.

- [ ] **Where** do fragments live per tool? *(Location decided: in `{tool}-template/`, co-located with partner; not under `metadata/`.)*

**Answer (Q5 ÔÇö merge behavior):** End goal: merge with the user's current config file **after having done a backup of the original file**. Installer must (1) back up the user's existing config, then (2) merge the fragment into/with the user's config (exact merge semantics ÔÇö overlay by key, concatenation, etc. ÔÇö remain per-tool). Document so installer behavior is unambiguous.

- [ ] How are fragments **merged** into the target tool's config per tool? *(Merge goal decided: backup first, then merge; exact semantics per-tool.)*
- [ ] **Deliverable:** Per-tool config-fragment contract (format, location, merge behavior) in a few sentences or a small example per tool. *(Done: summary and example below.)*

**Config-fragment contract (summary):** Config fragments use each tool's native format (no single format across tools). They live in `{tool}-template/`, co-located with the artifact they configure: tool-level config in the template root, skill-level config next to the skill in that template. The installer (1) backs up the user's existing config file, then (2) merges the fragment into it; exact merge semantics (overlay by key, concatenation, etc.) are per-tool.

**Example (minimal):**

| Tool   | Fragment path (repo)                          | Target (user)           | Format | Merge |
|--------|-----------------------------------------------|-------------------------|--------|--------|
| Codex  | `codex-template/config.toml`                  | `~/.codex/config.toml`  | TOML   | Backup target; merge fragment keys into user config (or write `.mdt.toml` when preserving user file). |
| Codex  | `codex-template/skills/ai-learning/config.json` | skill config at install dest | JSON | Backup then overlay by key. |
| Cursor | `cursor-template/...` (tool/skill config)    | tool config root / skill dir | per Cursor | Backup then merge; semantics per Cursor contract. |

Implementer: resolve exact target paths and merge rules from the tool support map or installer docs per tool.

### 6. Template-root add-on contract

**Answer (Q6 ÔÇö discovery):**

- **Layout:** Each `{tool}-template/` folder **mirrors the output structure** that will end up under `~/.{tool}/` (or wherever that tool needs its files).
- **Allowlist (fixed):** To avoid drift/mistakes, add-on dirs may contain only: **metadata** (`skill.meta.json`), **docs** (`SKILL.md`, `STYLE_PRESETS.md`), **config** (`config.json`), **agents/** (agent definitions), **rules/**; **no** full duplicate of `scripts/` or other runtime trees.
- **Enforcement:** Via **e2e testing** ÔÇö install to a tmp folder and assert what ended up there. The **installer** only complains if it couldn't find something it was expecting or if there were errors copying files to their desired destination; it does not validate allowlist at install time.

**Deliverable:** Add-on layout, fixed allowlist, and enforcement approach (e2e; installer fails only on missing expected or copy errors). Done.
