## Goal

- [ ] Find and document the answers to the open questions in the parent plan’s “define schemas and file locations” workstream so that the dependency sidecar schema, tool support-map schema, file layout, entrypoint criteria, and merge/add-on contracts can be locked and implemented without guesswork.

## Why This Plan

- [ ] The parent plan is high-level by design; its first workstream blocks all others (support maps, artifact metadata, resolver, first vertical slice).
- [ ] Answering these questions here produces a single discovery artifact that the parent plan can consume to complete workstream 1 and unblock implementation.

## Scope

- [ ] This plan is **discovery and documentation only**. Implementation of schemas, validators, or installer changes belongs in the parent plan.
- [ ] Out of scope: tool support map population, artifact migration, resolver logic, or ai-learning changes.

## Questions to Answer

### 1. Dependency sidecar schema

- [ ] What are the **field names and types** for the common dependency sidecar (e.g. `requires`, `optional`, `capabilities`, `tools`)?
- [ ] What **capability identifiers** exist? (Today the installer uses `hooks`, `runtimeScripts`, `sessionData`; the capability matrix uses `official`, `experimental`, `repo-adapter`, `unsupported`. How do these map into a single vocabulary?)
- [ ] Do dependencies express **version or constraint semantics**, or only presence?
- [ ] Should the schema include a **version field** for future compatibility?
- [ ] **Deliverable:** Schema sketch (e.g. JSON example) and a list of capability IDs that the support maps and package `requires` will use.

### 2. Tool support-map schema and file layout

- [ ] What is the **file layout** under `metadata/tools/`? (One file per tool? One combined file? How are baselines vs per-feature status split?)
- [ ] What is the **support-map schema**? (Field names for “always-installed baseline surfaces,” “supported artifact types,” “capability status by feature,” “error vs warn on mismatch.”)
- [ ] How does the schema align with the existing vocabulary in `docs/tools/capability-matrix.md` (e.g. official, experimental, repo-adapter, unsupported)?
- [ ] **Deliverable:** File naming convention, directory layout, and a minimal JSON schema or example for one tool.

### 3. Load-bearing and install-relevant entrypoints

- [ ] What **criteria** define “load-bearing” vs “install-relevant” runtime entrypoints? (e.g. “referenced by a package manifest, a hook, or a tool support map.”)
- [ ] Is there a **discovery rule** (e.g. “any script in package `tools.*.scripts` or invoked by a hook”) or an **enumerated list** for the first vertical slice?
- [ ] Which **exact entrypoints** in the first vertical slice (continuous-learning, ai-learning, Codex observer) must have sidecars? (List or rule so implementers know what to add.)
- [ ] **Deliverable:** Short definition or rule set, plus an explicit list of first-slice entrypoints that need sidecars.

### 4. Merge and precedence rules

- [ ] When **package manifest** and **artifact sidecar** both declare requires/optional, how do they combine? (Union? Override? Package wins for top-level, sidecar for transitive?)
- [ ] When a **tool support map** says a capability is unsupported and a **package** marks it required, is the outcome always “fail install” (map is authoritative for “can install,” package for “wanted”)?
- [ ] How do **template-root add-ons** merge with canonical source? (e.g. Codex overlay files overlay on top of shared `skills/ai-learning` — what is the precise rule?)
- [ ] **Deliverable:** Precedence and merge rules in short, actionable form (e.g. “required = union across closure; tool map authoritative for installability; package/sidecar authoritative for intent.”).

### 5. Config-fragment contract

- [ ] What is the **format** of a config fragment? (JSON, TOML, other? Key structure?)
- [ ] **Where** do fragments live? (Next to artifact? Under `metadata/`? Under template roots?)
- [ ] How are fragments **merged** into the target tool’s config? (Overlay by key? Concatenation? Document the intended behavior so the installer can implement it.)
- [ ] **Deliverable:** Fragment format, location rule, and merge behavior in a few sentences or a small example.

### 6. Template-root add-on contract

- [ ] What **directory layout** do add-ons follow in template roots? (e.g. `cursor-template/skills/<name>/` with only certain subdirs allowed.)
- [ ] What **may** appear in add-ons? (Metadata only? Docs? Config files? Allowlist of extensions or directory names.)
- [ ] How will “no duplicate runtime trees” be **enforced**? (Validator that fails on disallowed file types in add-on dirs? Installer check? Document the chosen approach.)
- [ ] **Deliverable:** Add-on layout, allowlist or rules, and enforcement approach (validator vs installer vs review).

## Suggested Approach

1. **Inventory current state** — List existing shapes: package `requires`, `skill.meta.json` (via `scripts/lib/skill-metadata.js`), `commands/*.meta.json`, and installer constants (`TARGET_CAPABILITIES`, `BASELINE_*`) so proposed schemas don’t invent unnecessary new concepts.
2. **Answer schema and layout first** — Questions 1 and 2 lock the vocabulary and file structure; they inform how entrypoints (3) and merge rules (4) are expressed.
3. **Define entrypoint criteria** — Question 3 can be answered with a rule plus a first-slice list; the list validates the rule.
4. **Document precedence** — Question 4 can be answered in parallel or after 1–2, using the schema vocabulary.
5. **Lock config and add-on contracts** — Questions 5 and 6 support the first vertical slice (e.g. Codex observer, ai-learning config defaults); answer with enough detail that the parent plan’s “first vertical slice” workstream can implement without ambiguity.
