# Composable MDT — Future Architectural Redesign

- Status: `not-started`
- Created: `2026-03-14`
- Last updated: `2026-03-14`
- Owner: `mdt`
- Execution mode: `mixed`
- Related docs: [20260314.10.37.ai-learning-cwd-and-process-model.md](20260314.10.37.ai-learning-cwd-and-process-model.md)

## Goal

Replace the current hybrid install layout (tool-facing surfaces at config root,
runtime dependencies nested under `mdt/`) with a **compose model** where:

1. Shared library code lives once in the repo
2. A deterministic compose step assembles self-contained skill packages
3. The composed layout is identical in dev staging and production installs
4. Relative paths in skill code work without fallback chains or candidate loops

This plan is a **design document for a future implementation**. It should not be
started until the immediate ai-learning fix (move code into the skill) is
complete and validated, and until a second skill actually needs shared library
code — giving a concrete reuse case to validate against.

## Problem Statement

The current install model has two layers that cross-reference each other at
runtime:

- **Layer 1** — tool-facing surfaces (`~/.claude/skills/`, `commands/`, etc.)
- **Layer 2** — MDT infrastructure (`~/.claude/mdt/scripts/lib/`)

Skills in Layer 1 `require` code from Layer 2 via multi-candidate path
resolution chains. This creates:

- Dead code in fallback chains (candidates that never resolve)
- Silent breaks when the install layout changes
- Different path resolution in repo vs installed context
- Bugs that pass tests but fail in production (or vice versa)

## The Compose Model

### Concept

A **compose step** reads a manifest per skill, copies shared library files into
the skill directory, and produces a self-contained output. The compose step runs:

- During `npm install` / `scripts/install-mdt.js` → production install
- During `npm pretest` → dev staging directory
- Both produce the **identical** directory structure

### Manifest schema (per skill)

```json
{
  "compose": {
    "sources": {
      "lib/": "lib/continuous-learning/"
    }
  }
}
```

Where:
- Key (`lib/`) is the destination inside the skill directory
- Value (`lib/continuous-learning/`) is the source relative to repo root

The compose step copies every file from source to destination. Directory
structure is preserved.

### Repo layout (unchanged)

```
lib/
  continuous-learning/
    project-detection.js
    observer-runtime.js
    codex-learn-runtime.js

skills/
  ai-learning/
    SKILL.md
    config.json
    compose.json              ← manifest
    scripts/
      detect-project.js       ← require('../lib/project-detection.js')
    agents/
    hooks/
    lib/                      ← EMPTY in repo, composed at build time
```

### Composed output (dev staging AND production install)

```
skills/ai-learning/
  SKILL.md
  config.json
  scripts/
    detect-project.js         ← require('../lib/project-detection.js')
  lib/
    project-detection.js      ← composed from lib/continuous-learning/
    observer-runtime.js
    codex-learn-runtime.js
  agents/
  hooks/
```

### Dev staging

```bash
npm run compose              # or runs automatically via pretest
```

Produces `.mdt-staging/` (gitignored) with the composed layout. Tests run
against this directory.

## Benefits

- **Dev = production.** The composed layout is identical in both contexts. A
  path that works in staging works after install. A path that breaks in staging
  breaks before commit.
- **Self-contained skills.** No runtime cross-directory `require` calls. A skill
  can be copied to any directory and it works.
- **Explicit dependencies.** The compose manifest is the auditable bill of
  materials per skill. `grep compose.json skills/*/` shows all shared library
  consumers.
- **Multi-tool install.** Compose to `~/.claude/skills/`, `~/.cursor/skills/`,
  `~/.codex/skills/` — same structure, no tool-specific path adjustments.
- **New tool onboarding.** Adding a Windsurf or Zed adapter is just "compose to
  a new root" — no code changes in skills.

## Risks

### 1. Build step in a zero-build repo

MDT is currently a zero-build Node.js project. Adding a mandatory compose step
changes the developer workflow. Every edit to shared library code requires
re-compose before the change is visible to a skill.

**Mitigation:** The compose step is pure file copies — sub-second on any machine.
Run it automatically in `pretest`. Consider a `--watch` mode for active
development.

### 2. Stale staging directory

If a developer runs tests without re-composing, they test against stale code.
Bugs introduced in the shared library won't surface until the next compose.

**Mitigation:** Always compose in `pretest`. Add a CI step that verifies staging
matches a fresh compose. Consider content-hashing the staging dir against source.

### 3. Manifest maintenance

If someone adds a file to the shared library but forgets to update the compose
manifest, the skill works in development (raw source is accessible at repo root)
but breaks after install (compose didn't include the new file).

**Mitigation:** The compose manifest should use glob patterns (`lib/continuous-learning/*`)
rather than individual file lists. New files are automatically included.
Alternatively, the compose step could warn if the skill's `lib/` directory is
empty after compose.

### 4. Windows symlink limitations

Symlinks require elevated privileges or developer mode on Windows. The compose
step must use file copies, not symlinks.

**Mitigation:** Always copy. Accept the sub-second cost. Do not offer a symlink
mode — the consistency is worth more than the performance.

### 5. Speculative infrastructure

Today only `ai-learning` needs shared library code. Building compose
infrastructure for a single consumer adds complexity without proven reuse.

**Mitigation:** Do not build this until a second skill actually needs shared
library code. The immediate fix (move code into the skill) solves the current
problem. This plan is a design-on-paper exercise.

## Verification Strategy

### Compose correctness

- [ ] Content-hash the composed staging dir and the production install dir. They
  must match byte-for-byte (excluding timestamps).
- [ ] Run `npm test` against the staging dir. All tests pass.
- [ ] Copy a single composed skill to a temp directory (outside the repo). Verify
  it loads and runs without errors.

### No cross-layer references

- [ ] `grep -r "require.*mdt/scripts" .mdt-staging/skills/` returns zero matches.
- [ ] `grep -r "__dirname.*\.\..*\.\..*\.\." .mdt-staging/skills/` returns zero
  matches (no 3+ level path traversals).
- [ ] Every `require` in a composed skill resolves to a file inside that skill's
  own directory tree.

### Installer parity

- [ ] Run `scripts/install-mdt.js` to a temp target. Compare the skills
  directory against `.mdt-staging/skills/`. They must be structurally identical.
- [ ] Run `diff -r .mdt-staging/skills/ /tmp/install-target/skills/` — no
  differences.

### Regression

- [ ] `npm test` passes in CI (against composed staging).
- [ ] `npm run lint` passes (compose artifacts are gitignored, no lint noise).
- [ ] Manual install to `~/.claude/` and verify `/ai-learning` workflow runs.

## Open Questions / To Investigate

- [ ] Should `compose.json` live in the skill directory (co-located) or in the
  package manifest (`packages/continuous-learning/package.json`)? Co-located is
  simpler but adds a file to every skill that uses shared code. Package-level
  keeps the skill clean but adds indirection.
- [ ] Should the staging directory be `.mdt-staging/` at repo root or inside
  `.artifacts/`? It should be gitignored either way.
- [ ] Should `lib/` in the repo be the shared library root, or should it stay at
  `scripts/lib/`? Renaming to `lib/` makes the compose manifest cleaner but is
  a large rename.
- [ ] How should per-tool driver files (claude-driver, cursor-driver, codex-driver)
  interact with compose? Are they shared library code or skill-specific code?
- [ ] Should compose support version pinning or is "always latest from repo" the
  right model for a monorepo?
- [ ] What happens to `scripts/lib/continuous-learning/` after compose is in
  place? Does it move to `lib/continuous-learning/` or stay where it is with
  compose manifests pointing to the current location?
- [ ] Is there value in a `compose --check` mode that verifies the staging dir
  is up to date without rebuilding? Useful for CI gates.

## Non-goals

- This plan does not cover the immediate ai-learning fix. That is tracked
  separately and should be completed first.
- This plan does not propose a JavaScript bundler or transpiler. Compose is pure
  file-copy — no code transformation.
- This plan does not change the tool-facing surface locations (`skills/`,
  `commands/`, `agents/`, `rules/`). Those stay at the tool's config root.
