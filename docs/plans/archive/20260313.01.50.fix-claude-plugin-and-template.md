## Goal

- [ ] Fix `plugin.json` to declare all MDT agents, commands, skills, and rules so plugin-based Claude Code installs deliver functional assets, and populate `claude-template/` so manual-install users have parity with the Cursor and Codex templates.

## Non-goals

- [ ] Do not add a `"hooks"` field to `plugin.json` (auto-loaded by convention; enforced by existing test).
- [ ] Do not copy the full `skills/` tree into `claude-template/` — the plugin path is the intended delivery mechanism for skills.
- [ ] Do not change the Cursor or Codex templates.
- [ ] Do not migrate scripts to `.mjs` (deferred backlog item).

## Open Questions / Assumptions

- [x] Plugin path resolution: `PLUGIN_SCHEMA_NOTES.md` known-good example uses `./agents/planner.md`, implying paths are repo-root-relative (not relative to `.claude-plugin/`). Verify during Phase 1 implementation before committing.
- [x] Whether `rules/` should also be declared in `plugin.json`. **Decision: yes, rules are intended to be installed via the plugin.**
- [x] `plugin.json` agents must be explicit file paths (NOT directory paths) per `PLUGIN_SCHEMA_NOTES.md`.

## Decisions

- [x] `agents` must be an array of 18 explicit `.md` file paths.
- [x] `commands`, `skills`, and `rules` may use directory paths wrapped in arrays: `["./commands/"]`, `["./skills/"]`, `["./rules/"]`.
- [x] `claude-template/` will receive: `CLAUDE.md`, `rules/` (common rules subset), and a core subset of commands.
- [x] Skills are NOT copied into `claude-template/` — plugin install is the skills delivery path.
- [x] A new completeness test guards `plugin.json` against regression to metadata-only.
